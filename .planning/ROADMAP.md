# AI Interview Assistant - Phase 7 Roadmap

**Project:** HRTech ATS - AI Interview Assistant Feature  
**Phase:** 7 (New Feature Implementation)  
**Timeline:** Estimated 4-6 weeks (coarse-grained execution)  
**Execution Model:** Parallel-enabled, research-backed, verification-gated  

---

## Executive Summary

This roadmap outlines the implementation of an AI-powered Interview Assistant that generates 3 personalized technical interview questions based on candidate skill gaps. The feature leverages existing Neo4j skill gap data and integrates OpenAI GPT-4o-mini while maintaining LGPD compliance and strict recruiter-only access.

**Core Value Proposition:**
- Accelerate interview prep: Recruiters generate targeted questions in seconds
- Data-driven questions: Grounded in candidate's actual skill gaps from Neo4j
- Secure & compliant: No PII leakage, LGPD audit trail, role-based access
- Reliable: Cached results, graceful degradation, 80% test coverage

## Phase Details

### Phase 1: Foundation & Permissions

**Goal:** Establish the persistence layer, permission boundaries, and Neo4j skill gap data access.

**Depends on:** Nothing (first phase of feature)

**Requirements:**
- REQ-01: InterviewQuestion Django model with audit fields (created_by, created_at)
- REQ-02: Neo4j skill gap queries to inform question generation
- REQ-05: Recruiter-only access (staff/admin roles)

**Success Criteria** (what must be TRUE when phase completes):

1. **Model exists & migrates cleanly**: `python manage.py migrate` succeeds with no errors; `InterviewQuestion` table created with columns: `id`, `candidate_id`, `questions` (JSONField), `created_by` (ForeignKey), `created_at` (timestamp), `updated_at`, `regenerated_count` (tracks regenerations)

2. **Neo4j queries return skill gaps**: Calling `fetch_candidate_skill_gaps(candidate_id)` returns a list of 2-5 skill gap objects with gap name, required level, and candidate proficiency. Tested against Neo4j test database.

3. **Permission layer enforces recruiter-only**: Attempting to access interview question endpoints without `staff=True` or `is_admin=True` returns 403 Forbidden. Django admin interface restricts question view/edit to admin users.

4. **Audit trail captured**: Every question creation logs `created_by=request.user` and `created_at=now()`. Django ORM query confirms two questions generated by different users have different audit fields.

**Deliverables:**

- `hrtech/models.py` → `InterviewQuestion` model (audit fields, candidate FK, JSON questions field)
- `hrtech/services/neo4j_skill_gap.py` → `fetch_candidate_skill_gaps()` function + unit tests
- `hrtech/permissions.py` → `RecruiterOnlyPermission` mixin for views
- Django migration file (auto-generated by `makemigrations`)
- Fixture with test candidate + Neo4j test data for local dev

**Key Dependencies:**
- Existing candidate model (FK target)
- Running Neo4j instance (test instance can be Docker-based)

**Risk Mitigation:**
- **Neo4j query complexity**: Start with simple skill gap retrieval; optimize later if slow
- **Migration backwards compatibility**: Always include rollback instructions in migration file

### Phase 2: Core Service Layer

**Goal:** Integrate OpenAI API, implement caching strategy, handle errors gracefully, and support edge cases.

**Depends on:** Phase 1 (needs `InterviewQuestion` model, Neo4j integration, permissions)

**Requirements:**
- REQ-03: OpenAI GPT-4o-mini integration with 15-second timeout
- REQ-04: Question caching in PostgreSQL (one generation per candidate unless regenerated)
- REQ-07: Robust error handling with graceful degradation
- REQ-08: Support for edge case: "no skill gaps" → Advanced Validation Questions

**Success Criteria** (what must be TRUE when phase completes):

1. **OpenAI API integration works end-to-end**: Calling `generate_interview_questions(candidate_id, skill_gaps)` returns a list of 3 well-formed questions (JSON: `{"question": "...", "category": "...", "difficulty": "..."}`). API calls respect 15-second timeout (no hanging requests).

2. **Caching prevents duplicate API calls**: Generating questions for the same candidate twice (without regeneration flag) returns cached results from PostgreSQL. Second call completes in <100ms (DB read only). No second OpenAI API call is made.

3. **Error handling is graceful**: If OpenAI API is down or timeout occurs:
   - Error is logged with context (candidate_id, timestamp, error message)
   - User sees friendly message: "Unable to generate questions. Try again in a moment."
   - No exceptions bubble to UI; request returns 500 with JSON error response
   - System continues functioning (other candidates unaffected)

4. **Edge case handled**: When candidate has no skill gaps, calling `generate_interview_questions(candidate_id, [])` returns 3 "Advanced Validation Questions" focused on system design, architecture, and domain knowledge (hardcoded fallback prompt).

5. **No PII leakage to OpenAI**: Candidate name, email, personal data never sent to OpenAI. Only skill gap names and difficulty levels in API request. Audit log shows what was sent.

**Deliverables:**

- `hrtech/services/openai_service.py` → `generate_interview_questions()` function with timeout handling, error logging
- `hrtech/services/question_cache.py` → `get_cached_questions()`, `save_questions_to_cache()` functions
- `hrtech/prompts/` → `QUESTION_GENERATION_PROMPT`, `NO_GAPS_FALLBACK_PROMPT` (constant strings with variables)
- Unit tests for OpenAI integration (mocked API calls)
- Error handling documentation (what happens when OpenAI is down)
- LGPD audit log entry for every OpenAI API call (input sanitization log)

**Key Dependencies:**
- OpenAI API key in `.env` (development credential)
- Existing Django settings for timeout configuration
- Phase 1 (Neo4j queries, permissions)

**Risk Mitigation:**
- **API cost overruns**: Implement rate limiting (max 5 question generations per candidate per day). Track API costs daily in logs.
- **Timeout failures**: Use Celery async task with retry logic; timeout at 15s, auto-retry up to 2x with exponential backoff.
- **PII leakage**: Code review checklist for data sent to OpenAI; automated test to verify no email/name in API payload.

### Phase 3: Frontend & User Workflows

**Goal:** Deliver the HTMX-powered UI for generating and regenerating interview questions on the candidate profile page.

**Depends on:** Phase 1 (permissions, model), Phase 2 (question generation service)

**Requirements:**
- REQ-06: HTMX inline UI update on candidate profile
- REQ-09: Question regeneration capability (overwrite old with new)

**Success Criteria** (what must be TRUE when phase completes):

1. **"Generate Interview Questions" button appears on candidate profile**: Button is visible only to users with `staff=True` or `is_admin=True`. Located in candidate sidebar (visible in <2 seconds on page load).

2. **Clicking button triggers question generation without page reload**: HTMX POST request to `/api/candidates/<id>/generate-interview-questions/` completes in <5 seconds (including OpenAI call). UI shows spinner while loading.

3. **Generated questions appear inline**: 3 questions display below button in a formatted card layout (question text, category, difficulty badge). Questions persist after page reload (read from `InterviewQuestion` model).

4. **Regeneration works with visual feedback**: Clicking "Regenerate" on existing questions shows confirmation dialog ("Replace current questions?"), then re-calls OpenAI API and updates UI inline. Old questions are overwritten in database (`regenerated_count` increments).

5. **Error states handled visually**: If generation fails, user sees friendly error message (e.g., "Unable to generate questions. Try again.") instead of blank state or 500 error page. No console errors visible.

**Deliverables:**

- `hrtech/views.py` → `generate_interview_questions` view (POST endpoint), `regenerate_interview_questions` view
- `hrtech/templates/candidate_profile.html` → Updated template with "Generate Interview Questions" button
- `hrtech/templates/interview_questions.html` → HTMX partial template for questions display
- `hrtech/static/js/interview_questions.js` → Event handlers for button click, regenerate confirm, spinner logic
- CSS for question cards and loading state (Tailwind or existing framework)

**Key Dependencies:**
- Phase 1 (permissions, model)
- Phase 2 (question generation service)
- Existing HTMX setup in project (already in use)

**Risk Mitigation:**
- **Network latency**: Use optimistic UI updates (show spinner immediately) or server sends progress events.
- **Double-click issue**: Disable button during request; re-enable only after response.
- **Mobile responsiveness**: Test question cards on mobile (ensure readable on small screens).

### Phase 4: Quality, Compliance & Deployment

**Goal:** Achieve 80% test coverage, pass LGPD audit, and harden the feature for production.

**Depends on:** Phases 1-3 (all feature code complete)

**Requirements:**
- REQ-10: 80% test coverage (unit + integration tests, all mocks)
- REQ-11: LGPD compliance (no PII to OpenAI)

**Success Criteria** (what must be TRUE when phase completes):

1. **80% test coverage achieved**: Running `pytest --cov=hrtech/services --cov=hrtech/views --cov-report=html` shows 80%+ coverage for interview assistant code. All critical paths covered (happy path, error cases, edge cases, edge case fallback).

2. **All OpenAI calls mocked in tests**: No real API calls made during test suite execution. All tests pass in <30 seconds. Mocked responses validate question structure (3 questions, JSON format, no PII in payload).

3. **LGPD audit trail complete**: Every OpenAI API call logged with:
   - Request timestamp, candidate_id (not name), skill gaps sent (no PII)
   - Response timestamp, question count returned
   - User who triggered generation (audit FK to User)
   - Logs queryable by `/admin/interview-questions/audit-log/`

4. **Data retention policy implemented**: `InterviewQuestion` records can be hard-deleted on user request. Django admin has "Delete questions" action. Deletion is logged in audit trail.

5. **LGPD documentation reviewed & signed off**: A brief LGPD compliance summary written (what data is sent where, retention period, deletion process). Reviewed by compliance/legal (internal sign-off).

6. **Integration test verifies end-to-end**: Test creates a candidate, generates questions via mocked OpenAI, checks DB cache, regenerates, verifies overwrite. All with mocked external API.

7. **Production checklist completed**: 
   - Environment variables documented (OPENAI_API_KEY, TIMEOUT settings)
   - Error logs reviewed (can find issues in Sentry/logs if OpenAI is down)
   - Database backups tested (restore and verify questions still readable)
   - Staging deployment tested (all endpoints return 200, questions generate with mocked API)

**Deliverables:**

- `tests/unit/test_openai_service.py` → 15-20 unit tests for question generation, caching, error handling (mocked API)
- `tests/integration/test_interview_questions_workflow.py` → 5-8 integration tests for full end-to-end flow
- `tests/fixtures/` → Reusable test data (candidate, Neo4j mock, OpenAI mock response)
- `docs/LGPD_COMPLIANCE.md` → 1-page summary of data flows, retention, deletion
- `docs/DEPLOYMENT_CHECKLIST.md` → Pre-production checklist (env vars, logging, DB backup)
- pytest configuration file (if not exists) with coverage thresholds
- Coverage report (HTML artifact)

**Key Dependencies:**
- Phases 1-3 (all feature code)
- Existing test framework (pytest, Django TestCase, etc.)
- Mocking library (unittest.mock, pytest-mock, responses for HTTP mocks)

**Risk Mitigation:**
- **Legacy code untested**: Start with new feature code only (don't aim for 80% of entire app).
- **Flaky tests**: Avoid hard-coded timeouts in tests; use async/await patterns or fixtures that wait for state.
- **LGPD over-documentation**: Keep compliance doc to 1 page; link to legal/security team for detailed review.

## Risk Matrix & Mitigation

| Risk | Severity | Mitigation | Owner |
|------|----------|-----------|-------|
| OpenAI API costs spike | Medium | Rate limit: 5 generations/candidate/day; daily cost alerts | Engineering |
| Timeout on slow Neo4j queries | Medium | Cache skill gaps in Redis (TTL 1h); async task with retry | Engineering |
| PII accidentally sent to OpenAI | High | Code review checklist; automated test for payload sanitization | Engineering + Compliance |
| Database schema migration fails | Low | Test migration in staging before prod; include rollback SQL in commit | Engineering |
| LGPD compliance gaps discovered late | Medium | Early compliance review at phase end (before prod); legal sign-off | Compliance |
| Test coverage falls below 80% | Low | Enforce coverage gate in CI (fail build if <80%); review weekly | Engineering |
